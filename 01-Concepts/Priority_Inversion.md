# Priority_Inversion（优先级翻转）

> 关联概念：[[Mutex]] | [[Thread]] | [[ThreadPriority]]
> 来源问题：[[05-Questions/优先级翻转如何发生？]]

## 本质

优先级翻转是指：**高优先级线程因为等待一个被低优先级线程持有的资源，而被一个与该资源无关的中等优先级线程间接延迟执行**的现象。最终效果是高优先级线程的实际响应时间反而比中等优先级线程更长——优先级"翻转"了。

## 解决什么问题

不是解决问题的概念，而是描述一个需要被解决的问题。理解它是为了知道何时用 Mutex（而非信号量），以及理解 Mutex 优先级继承机制存在的原因。

## 核心机制

**翻转如何发生（三线程经典场景）**：

```
线程优先级：H（高）> M（中）> L（低）

1. L 持有 Mutex，进入临界区
2. H 启动，尝试 lock Mutex → 被阻塞，等待 L 释放
3. M 启动（优先级高于 L）→ 抢占 L，开始长时间运行
4. 结果：H 必须等 M 跑完 → L 才能继续 → L 释放 Mutex → H 才能运行
   H 的等待时间 = M 的完整执行时间，与 Mutex 本身无关
```

**Zephyr 的解决方案——优先级继承**：

当 H 在等待 L 持有的 Mutex 时，内核自动将 L 的优先级临时提升至 H 的级别。此时 L 能抢占 M，尽快释放 Mutex，H 得以及时运行。L 释放 Mutex 后优先级自动恢复原值。

**为什么信号量不能解决这个问题**：信号量没有所有权，内核不知道"谁持有"，无法做优先级继承。这是需要用 Mutex 而不是信号量来保护共享资源的核心原因之一。

## 与相关概念的区别

| | Mutex | 信号量 |
|--|-------|--------|
| 优先级继承 | 有，自动处理翻转 | 无，存在翻转风险 |
| 适合保护共享资源 | 是 | 不推荐（有翻转风险） |

## 在 Zephyr 里怎么用

使用 `K_MUTEX_DEFINE` + `k_mutex_lock` / `k_mutex_unlock` 即可，Zephyr 内核自动启用优先级继承，无需额外配置。

## 坑

**现象**：系统响应延迟异常，高优先级任务明显滞后，但没有死锁。
**原因**：可能是优先级翻转。检查是否用信号量在保护共享资源——信号量没有优先级继承，中等优先级任务可以无限期延迟高优先级任务。
**解决**：将保护共享资源的信号量改为 Mutex。
