# Mutex（互斥锁）

> 关联概念：[[Semaphore]] | [[Thread]] | [[Priority_Inversion]]
> 来源问题：[[05-Questions/什么时候该用信号量而不是互斥锁？]]

## 本质

Mutex 是一个带**所有权（Ownership）**的内核同步对象。同一时刻只有一个线程能持有它，且必须由持有者解锁——这是它与信号量最根本的区别。

## 解决什么问题

保护共享资源的临界区：确保同一时间只有一个线程访问共享的硬件外设、全局变量或数据结构，防止数据竞争和状态损坏。

## 核心机制

**Lock**：若 Mutex 空闲 → 当前线程获得所有权，进入临界区。若已被占用 → 当前线程挂入等待队列，状态改为 `PENDING`，触发调度切走。

**Unlock**：释放所有权，若等待队列非空则唤醒队首线程，将所有权转交给它。

**优先级继承（Priority Inheritance）**：当高优先级线程在等待低优先级线程持有的 Mutex 时，内核临时将低优先级线程的优先级提升到等待者中最高的那个，避免优先级翻转导致高优先级线程被无关线程间接阻塞。锁释放后优先级自动恢复。

## 与相关概念的区别

| | Mutex | 信号量 |
|--|-------|--------|
| 所有权 | 有（谁锁谁解） | 无（任何线程/ISR 可 give） |
| 优先级继承 | 有 | 无 |
| 主要用途 | 保护共享资源 | 发通知、协调顺序、计数 |
| ISR 可用 | 不可（不能阻塞） | give 可以 |
| 递归加锁 | Zephyr 支持（需配置） | 不适用 |

**核心判断**：需要"谁拿谁还"的所有权语义，或需要优先级继承 → 用 Mutex。只需要"发个信号"或"数资源" → 用信号量。

## 在 Zephyr 里怎么用

```c
K_MUTEX_DEFINE(my_mutex);

k_mutex_lock(&my_mutex, K_FOREVER);
// 临界区：操作共享资源
k_mutex_unlock(&my_mutex);
```

**源码位置**：`kernel/mutex.c` → `z_impl_k_mutex_lock` / `z_impl_k_mutex_unlock`

## 坑

**现象**：在 ISR（中断）里调用 `k_mutex_lock` 导致系统崩溃。
**原因**：ISR 没有线程上下文，不能阻塞，Mutex 的等待机制依赖线程挂起，在 ISR 里调用是未定义行为。
**解决**：ISR 内保护共享资源改用 `k_spinlock` 或关中断（`irq_lock`）。
