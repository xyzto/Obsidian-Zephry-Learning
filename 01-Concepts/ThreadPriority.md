# ThreadPriority（线程优先级）

> 关联概念：[[Thread]] | [[Priority_Inversion]] | [[Mutex]]
> 来源问题：[[05-Questions/线程调度相关问题集]]

## 本质

线程优先级是内核调度器决定"谁先获得 CPU"的数字依据。Zephyr 中数字越小优先级越高，负数为协作式线程，非负数为抢占式线程。

## 解决什么问题

在多线程系统中，不同任务有不同的实时性要求。优先级机制让开发者显式表达"哪个任务更重要"，内核据此保证高实时性任务优先响应。

## 核心机制

**数值规则**：
- 负数（如 `-1`）：协作式优先级，不会被其他线程抢占，只在主动调用阻塞函数时切走
- `0` 及正数：抢占式优先级，更高优先级（更小数值）的线程就绪时立即抢占当前线程
- 数字越小 → 优先级越高（`-5` 高于 `0`，`0` 高于 `5`）

**抢占时机**：高优先级线程从 `WAITING` 变为 `READY` 的瞬间（如 `k_sem_give` 唤醒它），调度器立即中断当前线程，将 CPU 交给高优先级线程。

**时间片轮转（CONFIG_TIMESLICING）**：同优先级的多个线程之间，内核强制按时间片轮流执行，防止某线程长期独占 CPU。不同优先级之间不适用时间片，高优先级不阻塞则低优先级永远无法运行。

**空闲线程**：所有线程都在等待时，内核运行空闲线程（优先级最低），执行 `WFI` 进入低功耗等待，直到中断或定时器唤醒某个线程。

## 与相关概念的区别

| 场景 | 现象 | 原因 |
|------|------|------|
| 高优先级线程无 sleep | 低优先级线程饿死 | 抢占式线程不让出 CPU |
| 同优先级多线程 | 轮流执行 | 时间片轮转 |
| 高优先级等 Mutex | 可能翻转 | 见 [[Priority_Inversion]] |

## 在 Zephyr 里怎么用

```c
// 静态定义：优先级 5（抢占式）
K_THREAD_DEFINE(my_tid, 512, my_func, NULL, NULL, NULL, 5, 0, 0);

// 动态修改优先级（运行时）
k_thread_priority_set(k_current_get(), -1);  // 提升为协作式
```

**CONFIG_NUM_PREEMPT_PRIORITIES**：可抢占优先级数量，默认 16（0~15）。
**CONFIG_NUM_COOP_PRIORITIES**：协作式优先级数量，默认 16（-1~-16）。

## 坑

**现象**：高优先级线程设置后，低优先级线程完全不运行。
**原因**：高优先级线程没有任何阻塞调用（无 `k_sleep`、无 `k_sem_take`），永远不让出 CPU。
**解决**：高优先级线程在完成关键任务后必须调用阻塞函数让出 CPU，或将其设为协作式并手动调用 `k_yield()`。

---

**现象**：修改优先级数值后行为没变。
**原因**：增量编译缓存了旧配置。
**解决**：`west build -p always` 强制重新编译。
