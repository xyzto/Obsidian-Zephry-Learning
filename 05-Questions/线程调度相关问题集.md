## Q1：CPU 一直在执行线程吗？sleep 的时候去哪了？

**困惑：** 如果一个线程在 sleep，CPU 会停下来吗？

**结论：**
CPU 永远不会停。所有线程都在 sleep 时，内核运行**空闲线程（Idle Thread）**，执行 `WFI`（Wait For Interrupt）进入低功耗等待，直到下一个中断或定时器唤醒。

`k_msleep(3000)` 的本质：
1. 内核把当前线程状态改为 `Waiting`
2. 保存该线程的寄存器现场到堆栈
3. 扫描就绪队列，把 CPU 交给下一个线程

CPU 只是去干别的事了，不是停了。

---

## Q2：`k_sem_take` 是怎么让线程停下来的？

**困惑：** `k_sem_take` 只是个 C 函数，它怎么让代码"卡"在那一行？

**结论：**
不是"卡"，而是**主动触发了一次任务切换（Context Switch）**：

1. 检查信号量计数为 0
2. 把当前线程加入信号量的等待队列 `wait_q`
3. 把当前线程状态改为 `Pending`
4. 保存现场，把 CPU 交给其他就绪线程

当 `k_sem_give` 被调用时，内核从 `wait_q` 里取出等待的线程，恢复它的现场，从 `k_sem_take` 后面那行继续执行。

对开发者看起来像"停在那行"，对 CPU 来说是去执行别人了。

---

## Q3：`k_sem_take` 作用的是哪个线程？怎么知道的？

**困惑：** `k_sem_take` 在线程内部执行，它怎么知道要挂起"当前这个线程"？

**结论：**
内核维护一个全局指针 `_kernel.current`，始终指向当前占用 CPU 的线程控制块（TCB）。任何线程调用 `k_sem_take`，内核都直接操作这个指针指向的线程，不需要你声明"我是谁"。

## Q4：内核会一直轮询检查信号量吗？

**困惑：** 内核是不停地检查信号量的值吗？这不会浪费 CPU 吗？

**结论：**
**不会轮询**。内核是事件驱动的，只在两个瞬间工作：

- `k_sem_take` 被调用时 → 检查一次，没值就挂起线程，之后不再管
- `k_sem_give` 被调用时 → 查等待队列，把等待的线程唤醒

信号量就像门铃，不是闹钟。不按就不响，完全不占 CPU。

---

## Q5：信号量和线程之间的关系是谁维护的？

**困惑：** 两个线程怎么通过信号量关联起来的？这层关系存在哪？

**结论：**
信号量结构体本身包含一个**等待队列** `wait_q`（链表）：

```
struct k_sem {
    uint32_t count;      // 当前计数
    uint32_t limit;      // 最大计数
    wait_q_t wait_q;     // 等待的线程列表
}
```

- 你通过 `K_SEM_DEFINE` 在 RAM 里创建了这个结构体（含空队列）
- 线程 B 调用 `k_sem_take` 没拿到时，内核把 B 的 ID 写入 `wait_q`
- 线程 A 调用 `k_sem_give` 时，内核从 `wait_q` 里取出 B 并唤醒

两个线程不直接认识，都通过同一个信号量对象作为中介关联。

## Q6：优先级抢占是什么？高优先级线程会"插队"吗？

**困惑：** 把线程 B 设为 `-1` 优先级后，`k_sem_give` 之后谁先执行？

**结论：**
会立即抢占。执行顺序变为：

```
A: 准备发送信号...
B: 收到！立刻打印...   ← B 插队到 A 的中间
A: 信号已送达！
```

`k_sem_give` 触发内核唤醒 B 的瞬间，内核发现 B 优先级（-1）高于 A（0），立即保存 A 的现场，把 CPU 交给 B。B 执行完再归还。

**这就是为什么 RTOS 中优先级不能乱设**：优先级过高的线程如果不释放 CPU（不 sleep / 不阻塞），会导致低优先级线程永远得不到执行（线程饥饿）。
