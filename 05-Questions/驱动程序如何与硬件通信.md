# 驱动程序如何与硬件通信？

> 来源：Zephyr 学习中对"寄存器地址从哪来"的困惑
> 状态：#已验证
> 关联概念：[[DeviceTree]] | [[Overlay]] | [[Zephyr启动与设备初始化]]

## 当前理解

DTS 描述硬件，编译器把 DTS 转成 C 宏，驱动用宏拿到寄存器地址，然后直接操作寄存器。

## 拆解

### 1. 规则从哪来？—— Bindings（YAML 文件）

DTS 不是随便写的，每种硬件都有对应的规则文件：

```
zephyr/dts/bindings/gpio/gpio-leds.yaml
zephyr/dts/bindings/display/zephyr,sdl-dc.yaml
```

规则文件定义了：
- `compatible` 字符串必须叫什么
- 允许哪些属性（写了没定义的属性就报错）
- 属性的类型（数字、字符串、引用）

这就是为什么在 overlay 里写了未定义的属性编译器会报错——它在 YAML 里查不到。

### 2. 寄存器地址从哪来？—— 五层结构

```
厂家 Datasheet
    ↓ 写死物理地址（芯片出厂时固化在硅片上）
.dtsi 文件（Zephyr 官方维护）
    ↓ reg = <0x40010000 0x400>
board.dts（开发板级）
    ↓ 引用或覆盖
app.overlay（你写的）
    ↓ &uart1 { status = "okay"; }
编译器生成 devicetree_generated.h
    ↓ #define DT_UART1_BASE_ADDRESS 0x40010000
驱动代码
    ↓ struct regs *r = (struct regs *)DT_INST_REG_ADDR(0)
    ↓ r->DATA = value  → CPU 往物理地址发信号 → 硬件响应
```

### 3. QEMU 的特殊情况

QEMU（native_sim）没有真实寄存器，驱动内部用 C 结构体模拟寄存器，链接器随机分配内存地址，DTS 里不需要写死地址。

## 结论

```
你写 overlay 的 &uart1
    ↓ 引用的是 dtsi 里已有的节点（含寄存器地址）
编译器把地址转成 DT_* 宏
    ↓ 驱动用宏拿地址
驱动把地址强转为结构体指针
    ↓ 直接读写寄存器字段
硬件响应
```

DTS 是"胶水层"——把硬件的物理地址和软件的驱动代码解耦，换板子只改 DTS，驱动代码不动。
