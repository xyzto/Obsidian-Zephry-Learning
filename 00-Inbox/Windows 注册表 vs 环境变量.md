# Windows 注册表 vs 环境变量（严谨版)

---

## 一、核心结论

环境变量不是注册表的简单映射，而是一个“多来源合成的进程级运行环境结构”。

---

## 二、整体架构

```
【应用程序】
      ↑
【环境变量（进程内，运行时）】
      ↑
【多来源输入】
  ├─ 注册表（持久化配置）
  ├─ 父进程继承
  ├─ 运行时注入（set / API）
  └─ 登录脚本 / 组策略
```

---

## 三、注册表的角色（存储层）

本质：Windows 的统一配置数据库（持久化）

环境变量相关路径：

```
用户变量：
HKEY_CURRENT_USER\Environment

系统变量：
HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Session Manager\Environment
```

---

## 四、环境变量的本质（运行时结构）

本质：存在于进程内存中的环境参数表

特点：

- 每个进程一份（独立）
    
- 通过父进程继承
    
- 可在运行时修改（仅影响自身及子进程）
    

---

## 五、环境变量的来源

```
环境变量 =
  父进程继承
+ 注册表初始化（系统/用户变量）
+ 命令行 set（当前进程）
+ API（SetEnvironmentVariable）
+ 登录脚本 / 组策略
```

---

## 六、进程模型

### 继承机制

- 子进程复制父进程的环境变量表
    
- 形成独立副本
    

结论：

环境变量通过“进程树继承”传播

---

### 修改规则

|操作|影响范围|
|---|---|
|修改当前进程变量|当前进程 + 子进程|
|修改注册表变量|未来新进程|
|修改已运行进程|无效|

---

## 七、三层结构（优先级）

```
优先级（高 → 低）：

1. 进程级变量（临时）
2. 用户变量（HKCU）
3. 系统变量（HKLM）
```

规则：

高优先级覆盖低优先级

---

## 八、为什么使用环境变量而不是直接读注册表

原因：

1. 接口标准化（getenv / GetEnvironmentVariable）
    
2. 跨平台一致性
    
3. 权限隔离
    

结论：

环境变量是接口层，注册表是存储实现

---

## 九、PATH 修改不生效原因

原因：

进程已持有旧的环境变量副本

解决：

- 重开终端
    
- 重启程序
    

---

### 进阶机制

```
SendMessageTimeout(HWND_BROADCAST, WM_SETTINGCHANGE, 0, L"Environment")
```

说明：

通知系统环境变量已更新（是否生效取决于程序）

---

## 十、工程类比（嵌入式）

|Windows|嵌入式|
|---|---|
|注册表|Flash 配置区|
|环境变量|RAM 全局配置结构|
|进程变量|局部运行变量|

---

## 十一、最终总结

```
注册表：环境变量的主要持久化存储
环境变量：多来源合成的进程内运行环境

来源：
注册表 + 父进程继承 + 运行时注入

机制：
进程启动时继承 → 独立副本 → 仅影响自身

本质：
存储层 ≠ 使用层
```
