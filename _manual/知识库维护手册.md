# 知识库维护手册

> 适用于任何学科的 Obsidian + Git 知识工程系统
> 基于 Zephyr RTOS 学习仓库的实际整理流程提炼而成

**⚠ 阅读说明：** 这份文档是这套知识管理方法论的**完整设计记录**，具有通用性，可复用于任何学科。它描述的是系统的设计初衷和理想流程，不完全等同于当前仓库的实际操作方式（例如：本文第三节要求"必须先经过 Questions 再写 Concepts"，当前仓库已简化为学完直接写 Concept）。

当前仓库的具体操作规则以 [[AI协作提示词]] 为准。

---

## 一、核心理念

> **把笔记当成代码工程管理，而不是文档堆积。**

知识工程 = 问题驱动 + 结构化建模 + 实验验证 + 持续演化

三条不可违背的黄金法则：

| 法则 | 含义 | 类比 |
|------|------|------|
| **原子化** | 一个文件只讲一件事 | 函数单一职责 |
| **可链接** | 用 `[[]]` 建立概念网络，不用子文件夹区分层级 | 模块依赖注入 |
| **可演化** | 每次理解升级就 git commit，思维进化有迹可循 | 版本控制 |

## 二、目录结构

```
knowledge-base/
├── README.md          # 知识库总索引
├── .gitignore         # 至少排除 .obsidian/
├── _templates/        # 笔记模板（系统文件）
├── _manual/           # 仓库维护手册（系统文件）
│
├── 00-Inbox/          # 输入缓冲区：捕获，不处理，48h 内清空
├── 01-Concepts/       # 原子化概念库：扁平结构，禁止子文件夹
├── 02-Projects/       # 当前执行项目：每个项目一个子文件夹
├── 03-Areas/          # 长期关注领域：学科总览、学习进度追踪
├── 04-Resources/      # 参考资料：外部链接、教材、工具文档
├── 05-Questions/      # 问题驱动：每个问题一个文件
└── assets/            # 图片、附件
```

各区域角色：

| 区域 | 角色 | 修改频率 | 典型内容 |
|------|------|---------|---------|
| 00-Inbox | 缓冲区 | 每天写入，每周清空 | 零碎想法、摘抄、疑问 |
| 01-Concepts | 知识原子 | 高频，持续完善 | `DMA.md` / `CMake.md` |
| 02-Projects | 执行单元 | 项目期间高频 | 实验记录、训练计划 |
| 03-Areas | 领域导航 | 低频，只做索引 | `Zephyr-RTOS.md` 学习进度 |
| 04-Resources | 外部依赖 | 低频，只存引用 | 教材链接、工具手册 |
| 05-Questions | 问题引擎 | 高频，遇问题就写 | `为什么DMA比中断快.md` |

## 三、唯一主流程

```
捕获 → 提问 → 验证 → 沉淀 → 连接 → 迭代
```

| 触发条件 | 操作 |
|---------|------|
| 遇到不懂的东西 | 丢进 `00-Inbox`，文件名即问题，内容可以空着 |
| 整理 Inbox 时 | 转化为 `05-Questions` 中的问题笔记 |
| 查资料 / 做实验 | 在 `02-Projects` 里记录验证过程 |
| 理解成型后 | 提炼写入 `01-Concepts`，添加 `[[链接]]` |
| 理解有变化 | git commit 记录认知升级 |

**强制规则：不允许直接写 01-Concepts，必须先经过 05-Questions。**

---

## 四、每日最小执行（4步）

每天约 15 分钟，保持系统活跃：

1. **捕获**：看到不懂的 → 丢进 `00-Inbox`
2. **提问**：Inbox 内容 → 转化为 `05-Questions` 中的问题笔记，写下当前猜测
3. **验证**：查资料 / 写代码 / 做实验
4. **沉淀**：将结论写入 `01-Concepts`，建立链接，git commit

每周末额外 30 分钟：彻底清空 Inbox，把已验证的 Question 提炼为 Concept。

## 五、笔记质量标准

### Question 笔记（05-Questions/）

必须包含：当前猜测（不怕错）、拆解子问题、验证方式、结论、关联 Concept。

```markdown
# 为什么 DMA 比中断快？

## 问题来源
项目：USART 接收实验

## 当前猜测
DMA 不需要 CPU 参与搬运

## 拆解
- 谁在搬数据？
- CPU 在做什么？
- 总线由谁控制？

## 验证方式
- [ ] 查阅参考手册
- [ ] 编写对比实验

## 结论
DMA 通过总线控制器直接在外设和内存间搬运数据，CPU 只需配置，无需参与每次传输。

## 关联
[[DMA]] | [[中断机制]] | [[USART]]

## 状态
- [x] 已验证
```

### Concept 笔记（01-Concepts/）

每个 Concept 最终应包含：本质定义、核心机制、API 用法、与相关概念的区别、**踩过的坑**。

```markdown
# DMA

> 关联概念：[[中断机制]] | [[总线结构]] | [[USART]]

## 本质
外设 ↔ 内存 的直接数据搬运机制（绕过 CPU）

## 核心机制
- DMA 控制器 / 总线仲裁 / 触发源

## 对比
与 [[中断机制]] 的区别：中断需要 CPU 参与每次数据处理，DMA 不需要。

## 坑
**现象**：DMA 传输完成但数据错误
**原因**：Cache 一致性问题，CPU 读到了缓存里的旧数据
**解决**：传输前后调用 Cache 刷新函数

## 来源问题
[[为什么DMA比中断快？]]
```

### Area 笔记（03-Areas/）

只做导航，不写知识细节：

```markdown
# Zephyr RTOS

## 知识地图
- [[CMake]] | [[Kconfig]] | [[DeviceTree]]

## 当前进度
→ [[02-Projects/F103ZE/进度]]
```

## 六、命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 英文概念 | 大驼峰或全大写缩写 | `CMake.md` / `DMA.md` |
| 中文概念 | 概念名本身，不加后缀 | `中断机制.md` |
| 问题文件 | 疑问句，动词开头 | `为什么DMA比中断快.md` |
| 项目文件夹 | 全大写芯片型号或全小写描述 | `F103ZE/` / `QEMU/` |
| 禁止出现 | 序号、"未命名"、"总结" | ~~`1_笔记.md`~~ ~~`STM32总结.md`~~ |

## 七、Git Commit 规范

```bash
feat(concept):   新增 [概念名]
update(concept): 补充/重构 [概念名]
fix(concept):    修正 [概念名] 中的错误理解
link:            建立 [A] 与 [B] 的关联
merge(concept):  合并 [文件A] 与 [文件B]
move:            迁移 [文件名] 至 [目标目录]
rename:          重命名文件或目录
test(project):   验证 [实验名称]
clean:           删除已整合旧稿
```

## 八、系统三大禁忌

**禁忌 1：写总结文档**
```
❌ STM32学习总结.md
✅ DMA.md + 中断机制.md + USART.md（各自独立，相互链接）
```

**禁忌 2：用子文件夹替代链接**
```
❌ 01-Concepts/嵌入式/STM32/DMA.md
✅ 01-Concepts/DMA.md，文件内写 [[STM32]] [[总线结构]]
```

**禁忌 3：Inbox 堆积不清理**
```
超过一周未处理 → 系统失效
每周固定 30 分钟：Inbox → Questions → Concepts
```

- [ ] `00-Inbox` 是否有超过一周未处理的文件？

## 十、与 AI 协作整理仓库

知识库积累一段时间后，可用 AI 辅助做系统整理：

| 阶段 | 操作 |
|------|------|
| 扫描诊断 | 让 AI 读取目录树，识别散落文件、命名问题、重复碎片 |
| 设计规则 | AI 输出迁移映射表和合并策略，**执行前人工确认** |
| 读取内容 | AI 逐一读取源文件，判断合并/保留/存档/删除 |
| 执行整理 | AI 建目录、移文件、合并内容、生成新文件 |
| 收尾 | AI 生成 README 索引，人工确认后 git commit |

AI 不替你思考，但帮你减少结构维护成本。结论必须由你得出，AI 只帮你整理和提问。

## 十一、快速初始化新仓库

**PowerShell（Windows）**
```powershell
$name = "my-knowledge-base"
$dirs = "00-Inbox","01-Concepts","02-Projects","03-Areas","04-Resources","05-Questions","assets","_templates","_manual"
New-Item -ItemType Directory -Path $name
$dirs | ForEach-Object { New-Item -ItemType Directory -Path "$name\$_" }
Set-Location $name
git init
".obsidian/" | Out-File .gitignore -Encoding utf8
New-Item README.md
git add .
git commit -m "init: 建立知识库基础结构"
```

**Bash（Linux / Mac / WSL）**
```bash
name="my-knowledge-base"
mkdir -p $name/{00-Inbox,01-Concepts,02-Projects,03-Areas,04-Resources,05-Questions,assets,_templates,_manual}
cd $name && git init
echo ".obsidian/" > .gitignore
touch README.md
git add . && git commit -m "init: 建立知识库基础结构"
```
