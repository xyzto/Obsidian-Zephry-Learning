## 核心规则（必须遵守）

**1. 流程规则**
- 学到新概念 → 直接打开或新建 `01-Concepts/` 文件补充
- 遇到问题 → 新建 `05-Questions/` 文件，写猜测和拆解，搞懂后填结论
- 没有日记，没有中间层，两步完成

**2. 结构规则**
- `01-Concepts/` 扁平结构，禁止任何子文件夹
- 用 `[[链接]]` 建立概念关联，不用文件夹层级
- 每个项目文件夹统一包含：学习计划.md / 进度.md / 环境信息.md / 实验/

**3. 文件定位规则**

| 文件类型 | 放在哪里 |
|---------|---------|
| 知识概念 | `01-Concepts/` |
| 实验记录 | `02-Projects/<项目>/实验/` |
| 学科总览 | `03-Areas/` |
| 外部链接、参考资料 | `04-Resources/` |
| 尚未解决的问题 | `05-Questions/` |
| 操作手册、系统配置 | `_manual/` |

**4. Concept 文件推荐结构**
每个 Concept 最终应包含：本质定义、核心机制、API 用法、与相关概念的区别、**踩过的坑**（每次实验遇到报错或意外行为时回来补充）。

**5. 禁止行为**
- 禁止创建"学习总结"、"笔记汇总"类文件
- 禁止在 `01-Concepts/` 里建子文件夹
- 禁止同一件事在多个文件重复记录

## Git Commit 规范

```
feat(concept):   新增概念
update(concept): 补充/重构概念（常见：补充"踩过的坑"一节）
fix(concept):    修正错误理解
link:            建立概念关联
merge(concept):  合并文件
move:            迁移文件
test(project):   验证实验
clean:           删除旧稿
```

---

## 你每次操作前必须做的事

1. 用 `list_directory` 扫描相关目录，确认文件实际存在
2. 用 `read_text_file` 读取文件内容，不要凭记忆假设内容
3. 执行操作前告诉我你打算做什么，我确认后再执行
4. 操作完成后告诉我改动内容，以及建议的 git commit 命令

## 常用场景补充（按需追加在粘贴内容后）

**整理查重：**
```
今天的任务：扫描仓库目录，找出内容重复、位置错误、需要合并的文件，
给我一份分析报告，确认后逐步执行。
```

**新增概念：**
```
今天学了 [XXX]，帮我在 01-Concepts/ 里新建一个 Concept 文件，
或者找到应该补充进哪个已有文件。
```

**补充踩坑记录：**
```
我在做 [XXX] 实验时遇到了这个问题：[描述现象]，
原因是 [XXX]，解决方法是 [XXX]。
帮我把这条"踩过的坑"补充进 [Concept名].md。
```

**记录实验：**
```
我刚做完一个 [XXX] 实验，结果是 [XXX]，
帮我在 02-Projects/[项目名]/实验/ 里新建一个实验记录文件。
```

**清理 Inbox：**
```
00-Inbox 里有一些待处理文件，帮我判断每个文件应该
移入哪个目录、是否需要重写格式、是否可以直接删除。
```
