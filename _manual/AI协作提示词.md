## 核心规则

**流程**

```
00-Inbox（随手捕获）→ 05-Questions（整理成问题）→ 01-Concepts（沉淀为知识）
```

- 灵光一现 → 丢进 `00-Inbox/`，文件名就是想法，内容可以只有一行
- 想清楚了 → 从 Inbox 移到 `05-Questions/`，补上猜测和拆解，搞懂后填结论
- 学了新概念 → 直接新建或打开 `01-Concepts/` 文件补充
- 做完实验 → 在对应项目的 `实验/` 目录新建记录文件，进度.md 打勾
- 遇到报错或意外行为 → 回到对应 Concept 文件补一条 `## 坑`

**结构底线**
- `01-Concepts/` 严禁建子文件夹，所有 Concept 平铺
- 进度只在 `02-Projects/<项目名>/进度.md` 记录，不在其他文件重复
- 同一件事只存一个地方，禁止内容重复
- 禁止创建"总结"、"汇总"类文件

**Concept 文件结构**（新建时参考）
```
## 本质
## 解决什么问题
## 核心机制
## 与相关概念的区别
## 在 Zephyr 里怎么用
## 坑        ← 遇到报错时回来补，格式：现象/原因/解决
## 产生的问题
```

**新增项目**
参考 `_manual/新建项目模板说明.md`，或在 `02-Projects/` 下运行：
```powershell
.\new-project.ps1 <项目名>
```

## 操作规范

每次操作前：
1. 用 `list_directory` 或 `search_files` 确认目标文件实际存在
2. 用 `read_text_file` 读取内容，不靠记忆做判断
3. 告诉我你打算做什么，等我确认后再执行
4. 完成后说明改了哪些文件，给出 git commit 命令

**`_manual/` 里的文件不得主动修改，只在明确被要求时操作。**

## Git Commit 规范

```
feat(concept):   新增 [概念名]
update(concept): 补充/重构 [概念名]
fix(concept):    修正错误理解
link:            建立概念关联
test(project):   完成实验 [实验名]
merge:           合并文件
move:            迁移文件
rename:          重命名
clean:           删除旧稿
fix:             修复断链或错误路径
```
