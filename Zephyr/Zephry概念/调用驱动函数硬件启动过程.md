# 一、核心概念梳理

1. **Devicetree 描述硬件**
    
    - `.dts` 文件里描述了设备节点、地址、中断、属性等。
        
    - `bindings/` 定义每个设备类型的属性结构。
        
2. **DEVICE_DT_DEFINE 宏**
    
    - 驱动使用这个宏把设备对象注册到 Zephyr **特殊 linker section**（`.z_device`）。
        
    - 生成结构体 `struct device uart1_device = { ... }`。
        
    - 同时把 **驱动初始化函数** (`init`) 和 **初始化阶段** (`PRE_KERNEL_1/2`, `POST_KERNEL`) 绑定进去。
        
3. **系统启动遍历设备表**
    
    - `sys_init()` 或 `device_startup()` 遍历 `.z_device` section 的所有设备对象。
        
    - 对每个对象调用绑定在 DEVICE_DT_DEFINE 的 `init` 函数。
        
4. **真正启动硬件的函数**
    
    - 驱动的 `init` 函数通常定义在对应驱动源文件里，例如 `drivers/uart/uart_stm32.c` 的 `uart_stm32_init()`。
        
    - 这个函数里会调用 **底层 HAL / 寄存器操作函数**，配置 GPIO、时钟、寄存器，实现硬件启动。
        

---

# 二、完整调用链条示意

假设我们初始化 UART1：

```
board.dts（Devicetree）
      ↓
DEVICE_DT_DEFINE(DT_NODELABEL(uart1), uart_stm32_init, ...)
      ↓
生成 struct device uart1_device 放入 .z_device section
      ↓
系统启动时遍历 __device_start → __device_end
      ↓
调用 uart1_device->init(uart1_device)
      ↓
uart_stm32_init()
      ↓
调用 HAL / 直接寄存器操作
      ↓
UART 硬件启动完成
```

**重点**：真正操作硬件的是 **驱动里的 init 函数**，而 DEVICE_DT_DEFINE 只是把它注册到系统，系统在启动时统一调用。

# 一、从 Devicetree 到设备对象

1️⃣ **Devicetree 文件**（board.dts / overlay）

```dts
&uart1 {
    status = "okay";
    current-speed = <115200>;
};
```

- 描述设备类型、地址、配置参数
    
- 不涉及任何 C 代码执行
    

2️⃣ **编译期解析**

- `dtc` 将 `.dts` 转成 `.h` 文件，例如 `devicetree_generated.h`
    
- 为每个节点生成宏：
    

```c
#define DT_NODELABEL_UART1 0x1234
#define DT_UART_1_CURRENT_SPEED 115200
```

**工程思想**：把硬件描述转换成可被驱动读取的宏，而不是运行时动态解析。

---

# 二、驱动注册（DEVICE_DT_DEFINE）

在 `drivers/uart/uart_stm32.c` 中：

```c
DEVICE_DT_DEFINE(DT_NODELABEL(uart1),
                 uart_stm32_init,
                 NULL,
                 &uart1_data,
                 &uart1_cfg,
                 PRE_KERNEL_1,
                 CONFIG_KERNEL_INIT_PRIORITY_DEVICE,
                 &uart_stm32_driver_api);
```

- 生成 `struct device uart1_device`
    
- 把 `uart_stm32_init` 函数和 init 阶段、优先级绑定到这个对象
    
- 使用 `__attribute__((section(".z_device")))` 放入特殊 linker section
    

**关键点**：这里**并没有初始化硬件**，只是把 init 函数和设备信息注册到内核表里。

---

# 三、系统启动遍历设备表

在 `kernel/init.c` 中，核心逻辑：

```c
void z_sys_init(void)
{
    struct device *dev;

    for (dev = __device_start; dev < __device_end; dev++) {
        if (dev->init) {
            dev->init(dev);
        }
    }
}
```

解释：

- `__device_start` / `__device_end` → linker section `.z_device` 的起止
    
- 遍历每个注册的设备
    
- 调用它们绑定的 init 函数
    

**这一步真正启动了驱动，但仍然在 C 层，不直接操作硬件寄存器**。

---

# 四、驱动 init 执行（uart_stm32_init）

在 `drivers/uart/uart_stm32.c`：

```c
static int uart_stm32_init(const struct device *dev)
{
    const struct uart_stm32_config *cfg = dev->config;
    struct uart_stm32_data *data = dev->data;

    // 1. 启动时钟
    stm32_clock_enable(cfg->clock);

    // 2. 配置 GPIO 引脚
    stm32_pin_configure(cfg->tx_pin, ALT_FUNC);
    stm32_pin_configure(cfg->rx_pin, ALT_FUNC);

    // 3. 配置寄存器（波特率、模式等）
    LL_USART_SetBaudRate(cfg->uart, cfg->baud_rate);
    LL_USART_Enable(cfg->uart);

    return 0;
}
```

- **真正操作硬件**：配置时钟、GPIO、UART 寄存器
    
- 这一步才是 UART 真正“活起来”的地方

# 五、从 Devicetree 到硬件启动的完整链条

可以总结为：

```
1. Devicetree (.dts / overlay)
      ↓ (编译期生成宏)
2. DEVICE_DT_DEFINE 宏 (驱动源文件)
      ↓ (生成 struct device + init 函数注册到 .z_device section)
3. 系统启动 (z_sys_init → device_startup)
      ↓ (遍历设备表)
4. 驱动 init 函数执行 (uart_stm32_init)
      ↓ (调用 HAL / 寄存器操作)
5. 硬件被启动和配置完成
```

**关键理解点**：

- Devicetree → 描述硬件
    
- DEVICE_DT_DEFINE → 注册驱动和 init 函数
    
- 系统启动 → 遍历注册表，调用 init
    
- init 函数 → 真正启动硬件
    

Zephyr 的“自动初始化”就是 **把 init 调用统一调度**，并隐藏在 linker section + 遍历循环里，但底层驱动 init 是具体执行硬件配置的地方。
