### 1. Zephyr 线程的基本概念

在裸机编程中，你的代码通常是一个巨大的 `while(1)`。但在 Zephyr 中，你可以把程序拆分成多个独立的、看起来是“同时运行”的任务。

- **什么是线程？** 一个独立的函数，拥有自己的**栈空间（Stack）**和**优先级**。
    
- **并发的真相：** 在单核 CPU（如我们模拟的 Cortex-M3）上，多个线程并不是真正的物理并行，而是内核在极短的时间内（毫秒级）来回切换，让你感觉它们在同时运行。
    

### 2. 线程的生命周期与状态转换

线程就像一个员工，它在公司里有几种状态：

1. **就绪 (Ready)**：员工在位子上，活儿都准备好了，只要老板（内核调度器）一点名，马上就能干。
    
2. **运行 (Running)**：正在干活。
    
3. **等待/阻塞 (Unready/Waiting)**：在等快递（信号量）或者在睡觉（`k_msleep`），暂时不占用办公资源。
    
4. **终止 (Terminated)**：活儿干完了，离职了。
    

### 3. 创建线程：`k_thread_create` vs `K_THREAD_DEFINE`

Zephyr 提供两种创建方式，**强烈建议先掌握“静态定义”**，因为它最安全、最符合工业习惯。

#### A. 静态定义（推荐）

就像在编译时就给员工分配好了工位：

C

```
K_THREAD_DEFINE(id, stack_size, entry_func, p1, p2, p3, prio, options, delay);
```

#### B. 动态创建（更灵活）

在程序运行过程中临时招人：

C

```
k_thread_create(&my_thread_data, my_stack_area, stack_size, entry_func, ...);
```

### 4. 线程优先级 (Priority)

这是调度器的“点名标准”：

- **数字越小，优先级越高**（0 比 7 高）。
    
- **协作式 (Cooperative) 优先级**：负数（-15 到 -1）。一旦运行，除非它自己愿意休息（调用 sleep），否则别人抢不走 CPU。
    
- **抢占式 (Preemptible) 优先级**：正数（0 到 15）。高优先级的人随时可以把低优先级的人踢下去。

### 5. 今日实战：双线程协作实验

请修改你的 `src/main.c`，我们将创建一个高频闪灯线程和一个低频打印线程。

C

```
#include <zephyr/kernel.h>
#include <zephyr/drivers/gpio.h>

/* 1. 定义栈空间大小 */
#define STACK_SIZE 1024

/* 2. 获取设备树中的 LED（沿用你昨天的 Pin 7） */
static const struct gpio_dt_spec led = GPIO_DT_SPEC_GET(DT_ALIAS(led_status), gpios);

/* 3. 线程 A 的逻辑：快速闪灯 */
void blink_fast(void *p1, void *p2, void *p3)
{
    if (!device_is_ready(led.port)) return;
    gpio_pin_configure_dt(&led, GPIO_OUTPUT_ACTIVE);

    while (1) {
        gpio_pin_toggle_dt(&led);
        // 快速闪烁：100ms
        k_msleep(100); 
    }
}

/* 4. 线程 B 的逻辑：慢速打印 */
void log_info(void *p1, void *p2, void *p3)
{
    while (1) {
        printk("--- [线程B] 我在后台监控状态... ---\n");
        // 每 2 秒打印一次
        k_msleep(2000); 
    }
}

/* 5. 静态定义两个线程 */
// 优先级 7，由于是正数，它们都是“抢占式”的
K_THREAD_DEFINE(blink_tid, STACK_SIZE, blink_fast, NULL, NULL, NULL, 7, 0, 0);
K_THREAD_DEFINE(log_tid,   STACK_SIZE, log_info,   NULL, NULL, NULL, 7, 0, 0);

/* main 也是一个线程 */
int main(void)
{
    printk("系统启动，多线程开始工作...\n");
    return 0;
}
```

### 1. 为什么要有 Overlay？（解决“变”与“不变”）

想象你在写一个手机 App。如果你把“屏幕是 6.1 寸”写死在程序里，换个 6.7 寸的手机你就得重写代码。

- **`main.c` 是“不变”的逻辑**：无论什么板子，逻辑都是“开灯 -> 等待 -> 关灯”。
    
- **`.overlay` 是“变”的硬件**：今天灯在 Pin 7，明天在 Pin 5；今天经理叫 `gpio0`，明天叫 `gpioa`。
    

**掌握点**：当你看到空白文件夹，先写 `main.c`（逻辑），报错说找不到硬件时，再去写 `.overlay`（地图）来填坑。这种**“按需填坑”**的感觉就是掌握。

---

### 2. 为什么要有那 5 行语法？（解决“谁是谁”）

你最纠结的那个 `my_led_label: led_0 { ... };` 其实是在建立**映射链条**。

请在脑子里跑通这条线：

1. **C 代码** 拿着一张条子喊：“谁是 `led-status`？” （`DT_ALIAS`）
    
2. **`aliases`** 柜台查表说：“`led-status` 对应 `&my_led_label`。”
    
3. **`my_led_label`** 站出来说：“我在这，我代表 `led_0` 这个房间。”
    
4. **`led_0`** 房间门口挂着牌子：“我是 `gpio-leds` 类型的，归 `gpio0` 管。”
    

**掌握点**：如果你改了任何一个环节，整条线就断了。**报错就是你的老师**，看报错信息里提示哪个名字找不到，你就知道链条断在哪了。

---

### 3. 为什么多线程不打架？（解决“时间切片”）

你看到的 A 和 B 交替打印，其实是 CPU 在做**极速瞬移**。

- **`k_msleep` 是核心**：它不是简单的延时，它是**“自愿交权”**。
    
- 当 A 运行到 `k_msleep(1000)`，它就从 CPU 上“下台”去休息了。
    
- 此时 CPU 空了，内核（Scheduler）立刻查表：“谁还醒着？” 发现 B 醒着，于是 B “上台”。
    

**掌握点**：在 RTOS 里，**永远不要写死循环的空等待**（比如 `while(1);` 里面什么都不干）。必须给每个线程加个 `sleep`，否则优先级高的线程会“霸占” CPU，导致其他线程饿死。

### 💡 终极掌握测试：三个“如果”

如果你能回答这三个问题，你就真的掌握了：

1. **如果**我在 `prj.conf` 里把 `CONFIG_GPIO=y` 删了，程序还能动吗？（提示：驱动“合同”毁了）
    
2. **如果**我有两个 LED，我的 Overlay 应该增加什么？（提示：`aliases` 里加个新外号，`leds` 盒子里加个新零件）
    
3. **如果**我把线程 B 的优先级从 `7` 改成 `-1`（负数在 Zephyr 里是协作式高优先级），而 B 里面又没写 `k_sleep`，线程 A 还会跑吗？（提示：B 会霸占 CPU 直到死）
